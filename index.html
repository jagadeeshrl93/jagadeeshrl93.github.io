Case study

RAG Cloud Docs — deep dive

A production-grade RAG system built from scratch. No framework magic — every layer understood and owned.

The problem

On-call engineers spent 20+ minutes searching through hundreds of pages of runbooks during incidents. Keyword search returned too many irrelevant results. The cost of slow answers was real — extended outages.

The approach

Build a RAG system that answers in natural language, cites its source, and never hallucinates. Run it locally with Ollama so there are zero API costs and zero data leaves the network.

The result

Sub-second query latency. Faithfulness score of 0.929 across 28 domain-specific Q&A pairs. Context recall of 1.000 — the right chunks found every time. Runs entirely on local hardware.

Architecture
Documents → Loader → Chunker (512 tok) → Embedder → ChromaDB
Question → [Vector Search + BM25] → RRF Fusion → Top-5 Chunks → LLM → Answer + Citations
Key engineering decisions
Why hybrid retrieval over pure vector search?

Vector search understands meaning but fails on exact terms — flag names like --kubelet-timeout, error codes, version numbers. BM25 catches these precisely. Combining both with Reciprocal Rank Fusion improved context recall from 0.71 to 1.000.

Why build the chunker from scratch?

Framework chunkers hide the decision. Chunk size directly affects answer quality — too small and a step loses its context, too large and irrelevant noise enters the prompt. Tested 256, 512, and 1024 tokens. 512 with 50-token overlap gave the best faithfulness scores.

Why Ollama instead of OpenAI?

Zero cost, zero latency overhead from API calls, and no data leaves the network. For infrastructure runbooks containing sensitive config, this matters. The system auto-falls back to OpenAI if Ollama isn't running — best of both worlds.

Why RAGAS evaluation?

You can't improve what you can't measure. Every change to chunk size, retrieval strategy, or prompt was validated against 28 hand-written Q&A pairs. This caught a faithfulness regression when switching chunk sizes — saved hours of manual testing.
